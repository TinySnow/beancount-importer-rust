# 架构设计

## 1. 总体架构

本项目采用“解析-归一化-规则匹配-交易生成-文本输出”的流水线架构：

```text
原始文件(CSV/TSV/Excel文本导出)
  -> CsvRecordReader
  -> RawRecord（归一化中间模型）
  -> RuleEngine（全局规则 + provider 规则）
  -> Provider::transform
  -> Transaction
  -> BeancountWriter
  -> .beancount 文件
```

## 2. 核心模块职责

### 2.1 runtime

- `runtime/config_loader.rs`
  - 负责加载 global/provider/mapping 配置
  - 处理相对路径、fallback 路径、配置合并
- `runtime/mod.rs`
  - 负责导入总流程编排
  - 负责 provider 选择、转换统计、输出

### 2.2 model

- `model/data`：`RawRecord` 中间态
- `model/rule`：规则定义、条件匹配、结果聚合
- `model/transaction`：Beancount 交易模型
- `model/writer`：Beancount 文本序列化
- `model/mapping`：源列到标准字段映射
- `model/config`：配置模型

### 2.3 providers

- `interface::Provider` 定义扩展接口
- `providers/common.rs` 提供通用现金流交易转换逻辑
- `providers/securities/*` 处理证券类特殊记账逻辑

### 2.4 utils

- 编码识别与解码
- 金额清洗与变换（`abs` 等）
- metadata key 规范化

## 3. 规则系统设计

### 3.1 匹配顺序

规则按以下顺序排序后执行：

1. `priority` 升序
2. `specificity`（条件更多的后应用）
3. 文件顺序（后定义可覆盖前定义）

### 3.2 聚合策略

`MatchResult` 采用“覆盖式合并”：

- 标量字段：后命中覆盖先命中
- tags/links：累积
- metadata：同 key 后命中覆盖先命中

### 3.3 terminal 机制

- `terminal` 默认 `false`
- 命中 `terminal: true` 后立即停止后续规则匹配

## 4. metadata 设计

### 4.1 目标

- 对齐迁移项目的命名约定
- 保证生成 key 满足 Beancount 语法

### 4.2 归一化策略

1. 常见中文 key 映射到约定英文 key
2. 兼容老 key（如 `counterparty`）
3. 兜底转成 lowerCamelCase
4. 无法转换时生成稳定 hash key（`meta_xxxxxxxx`）

## 5. 证券建模说明

### 5.1 基本原则

- 买入：持仓增加，现金减少
- 卖出：持仓减少，现金增加，追加 PnL 占位行
- 手续费：入 `fee_account`
- 尾差：入 `rounding_account`

### 5.2 回购（逆回购）建模

- 使用固定面值建模 repo 持仓
- 开仓与到期分别记账
- 利息/损失通过收入/费用账户表达

## 6. 扩展性设计

新增 provider 只需：

1. 实现 `Provider` trait
2. 注册到全局 registry
3. 提供对应配置与映射文件

避免硬编码表头与业务字段，所有映射尽量放到 `src/mapping` 与 `src/config`。
