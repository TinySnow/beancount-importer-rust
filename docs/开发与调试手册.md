# 开发与调试手册

## 1. 建议开发流程

1. 修改代码或配置。
2. 运行 `cargo fmt`。
3. 运行 `cargo test --quiet`。
4. 用真实或脱敏样本跑一次 `cargo run`。
5. 用 Beancount 工具链做语法与平账检查（可选但推荐）。

## 2. 常用命令

### 2.1 查看 CLI 参数

```bash
cargo run -- --help
```

### 2.2 用白盒样本做快速回归

```bash
cargo run -- --provider alipay --source testsets/支付宝交易明细测试数据集.csv --config config/alipay.yml --output tmp/output/dev-alipay.beancount --log-level info
cargo run -- --provider wechat --source testsets/微信支付账单测试数据集.csv --config config/wechat.yml --output tmp/output/dev-wechat.beancount --log-level info
cargo run -- --provider jd --source testsets/京东交易流水测试数据集.csv --config config/jd.yml --output tmp/output/dev-jd.beancount --log-level info
cargo run -- --provider mt --source testsets/美团账单测试数据集.csv --config config/mt.yml --output tmp/output/dev-mt.beancount --log-level info
cargo run -- --provider icbc --source testsets/工商银行交易明细测试数据集.csv --config config/icbc.yml --output tmp/output/dev-icbc.beancount --log-level info
cargo run -- --provider ccb --source testsets/建设银行交易明细测试数据集.csv --config config/ccb.yml --output tmp/output/dev-ccb.beancount --log-level info
```

### 2.3 严格模式回归

```bash
cargo run -- --provider alipay --source testsets/支付宝交易明细测试数据集.csv --config config/alipay.yml --strict --log-level info
```

说明：`--strict` 下，任意一条记录解析或转换失败都会立即退出。

## 3. 日志级别建议

- 日常使用：`--log-level info`
- 定位问题：`--log-level debug`
- 深入追踪：`--log-level trace`
- 只看错误：`-q` 或 `--log-level error`

建议重点关注日志行：
- `CSV/XLSX headers`
- `parsing complete`
- `Transformation complete: success / ignored / failed`

## 4. 典型问题排查

### 4.1 表头不匹配

现象：有解析结果但字段大量为空。

排查：
1. 看日志中的 headers 是否和 mapping 列名一致。
2. 检查 `skip_header_lines` 与 `has_csv_header`。
3. 检查 `mapping_file` 实际加载路径。

### 4.2 编码导致乱码

现象：中文字段出现乱码，规则命中异常。

排查：
1. 配置 `csv_options.encoding` 为 `auto`、`UTF-8` 或 `GBK`。
2. 看日志中的“检测到编码”。

### 4.3 strict 与非 strict 行为差异

- 非 strict（默认）：坏行会告警并跳过，流程继续。
- strict：坏行会直接失败退出，适合 CI 把关。

### 4.4 卖出证券分录看不懂

这是常见误解。卖出时多出的 `pnl_account` 行是“已实现盈亏承接行”，不是额外手续费。

## 5. 测试建议

### 5.1 单元测试重点

- 规则条件匹配（字符串、正则、数值）。
- metadata key 归一化。
- 金额解析与 transform。
- Writer 输出语法（含 `commodity/open`）。

### 5.2 集成测试重点

- 不同 provider 的真实样本。
- 含说明头、多编码、列数不齐等边界输入。
- strict 与非 strict 两种模式。
