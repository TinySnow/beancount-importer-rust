# 供应商扩展指南

本文档说明如何基于当前工程结构新增一个 provider。

## 1. 选择目录

按业务类型放置文件：
- 银行：`src/providers/banks/<name>.rs`
- 第三方支付：`src/providers/third_party/<name>.rs`
- 证券：`src/providers/securities/<name>.rs`

## 2. 实现 `Provider` trait

### 2.1 现金流类 provider（推荐复用共享逻辑）

```rust
use crate::{
    error::ImporterResult,
    interface::provider::Provider,
    model::{
        config::provider::ProviderConfig,
        data::raw_record::RawRecord,
        rule::rule_engine::RuleEngine,
        transaction::Transaction,
    },
    providers::shared::{CashflowTransformOptions, transform_cashflow_record},
};

const XXX_OPTIONS: CashflowTransformOptions = CashflowTransformOptions {
    provider_name: "xxx",
    default_asset_fallback: "Assets:XXX",
};

pub struct XxxProvider;

impl Provider for XxxProvider {
    fn name(&self) -> &'static str {
        "xxx"
    }

    fn description(&self) -> &'static str {
        "XXX statement importer"
    }

    fn transform(
        &self,
        record: RawRecord,
        rule_engine: &RuleEngine,
        config: &ProviderConfig,
    ) -> ImporterResult<Option<Transaction>> {
        transform_cashflow_record(XXX_OPTIONS, record, rule_engine, config)
    }
}
```

### 2.2 证券类 provider（复用证券共享逻辑）

参考 `src/providers/securities/futu.rs` 或 `src/providers/securities/yinhe.rs`，
通过 `SecurityTransformOptions + transform_security_record` 实现。

### 2.3 关于 `parse`

`Provider` trait 的 `parse` 签名含 `strict_mode`：

```rust
fn parse(
    &self,
    path: &Path,
    mapping: &FieldMapping,
    config: &ProviderConfig,
    strict_mode: bool,
) -> ImporterResult<Vec<RawRecord>>
```

大多数场景可直接使用 trait 默认实现（内部已接入 `CsvRecordReader`）。
只有在你需要自定义读取流程时才覆盖它。

## 3. 注册 provider

在 `src/providers/mod.rs` 中注册：

```rust
registry.register(Arc::new(third_party::xxx::XxxProvider));
```

并在对应 `mod.rs` 暴露模块。

## 4. 增加配置文件

至少新增两份文件：
- `config/<name>.yml`
- `mapping/<name>.yml`

建议原则：
- Rust 代码不硬编码表头名称。
- 业务规则尽量放在 YAML `rules`。
- 账户命名沿用现有层级风格，避免随意改层级。

证券 provider 额外建议：
- 优先使用 `securities_accounts` 子结构配置证券账户：
  - `cash_account`
  - `fee_account`
  - `pnl_account`
  - `rounding_account`
  - `repo_interest_account`
- 旧字段 `default_cash_account/default_fee_account/default_pnl_account/default_rounding_account/default_repo_interest_account` 仍兼容，但建议仅用于历史配置迁移。

## 5. 最小验证步骤

1. 先跑 `--help` 确认参数正常。
2. 用一份最小样本验证可读可写：

```bash
cargo run -- --provider xxx --source <sample.csv> --config config/xxx.yml --output tmp/output/xxx.beancount --log-level info
```

3. 开启 strict 模式再跑一遍：

```bash
cargo run -- --provider xxx --source <sample.csv> --config config/xxx.yml --strict --log-level info
```

4. 观察日志：
- headers 是否匹配
- `parsing complete`
- `Transformation complete`

## 6. 建议补充测试

- 关键规则是否命中。
- metadata key 是否符合预期。
- 输出分录是否可被 Beancount 正常解析。
- 证券 provider 额外验证 `fee/pnl/rounding` 分录行为。

## 7. 提交前检查

```bash
cargo fmt
cargo test --quiet
```

同时更新文档：
- `README.md` 的支持 provider 列表。
- `docs/配置详解.md`（如新增了规则能力或配置字段）。
