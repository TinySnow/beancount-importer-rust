# 配置详解

本文档给出项目配置的完整语义说明，重点回答：
- 配置从哪里加载，谁覆盖谁。
- 规则如何匹配、如何合并、如何停止。
- 证券场景下 `fee/pnl/rounding` 等账户到底怎么落账。

## 1. 配置加载与覆盖顺序

导入时按以下顺序解析：
1. 全局配置 `--global-config`（未传参时尝试 `config/global.yml`、`src/config/global.yml`）。
2. provider 配置 `--config`。
3. 若 `--config` 文件不存在，回退 `config/<provider>.yml`、`src/config/<provider>.yml`。
4. 若仍找不到，会尝试 `global.yml` 中的 `providers.<provider>` 子配置。
5. 字段映射优先读取 `mapping_file`；未指定则回退：
   - `mapping/<provider>.yml`
   - `mappings/<provider>.yml`
   - `src/mapping/<provider>.yml`

补充说明：
- `provider` 配置会覆盖 `global` 默认值。
- provider 未设置的默认字段会回退到 global。
- `mapping_file` 相对路径会先按 provider 配置文件目录解析，再按当前工作目录解析。

## 2. provider 配置结构

典型文件：`src/config/<provider>.yml`

```yaml
name: "wechat"
mapping_file: "src/mapping/wechat.yml"

default_asset_account: "Assets:Wallet:WeChat:Balance"
default_expense_account: "Expenses:Unknown"
default_income_account: "Income:Unknown"
default_currency: "CNY"

# 证券类可选
default_fee_account: "Expenses:Broker:Fee"
default_pnl_account: "Income:Broker:PnL"
default_rounding_account: "Expenses:Broker:Rounding"

csv_options:
  delimiter: ","
  quote: '"'
  flexible: true
  encoding: "auto"
  comment: "#"

skip_header_lines: 0
has_csv_header: true

output:
  date_format: "%Y-%m-%d"
  decimal_places: 2
  account_prefix: null
  emit_open_directives: false
  open_date: null

rules: []
```

### 2.1 关键字段

- `name`：provider 显示名（会影响输出 `source` metadata）。
- `mapping_file`：字段映射文件路径。
- `default_*_account`：规则未指定账户时的默认账户。
- `default_currency`：无币种时默认币种。
- `skip_header_lines`：跳过头部说明行。
- `has_csv_header`：是否包含表头。

## 3. `csv_options`

支持项：
- `delimiter`：分隔符（默认 `,`）。
- `quote`：引号字符（默认 `"`）。
- `flexible`：是否允许行列不齐（默认 `false`）。
- `encoding`：编码名（如 `auto`、`UTF-8`、`GBK`）。
- `comment`：注释行前缀（可选）。

### 3.1 `flexible` 与 `strict` 的关系

- `flexible=false`：CSV 解析阶段遇到列数异常会直接报错。
- `flexible=true`：CSV 解析阶段可继续；
  - 非 strict：会记 warning 并跳过坏行。
  - strict：遇到坏行仍会失败退出。

## 4. output

支持项：
- `date_format`：交易日期输出格式。
- `decimal_places`：金额小数位。
- `account_prefix`：统一账户前缀（可选）。
- `emit_open_directives`：是否输出 `open` 指令。
- `open_date`：`open` 日期；不填时自动使用最早交易日。

输出行为补充：
- writer 会自动输出 `commodity`（非货币商品且涉及持仓/价格时）。
- metadata 会按 key 排序输出，保证结果稳定。

## 5. 规则系统

## 5.1 规则定义

```yaml
- name: "餐饮支出"
  match_mode: and
  conditions:
    - field: "type"
      equals: "支出"
    - field: "item"
      regex: "(咖啡|奶茶|外卖|餐饮)"
  action:
    debit_account: "Expenses:Food"
    credit_account: "Assets:Wallet:WeChat:Balance"
    metadata:
      scene: "daily"
    tags: ["food"]
    links: ["rule/food"]
    flag: "*"
  priority: 10
  terminal: false
```

## 5.2 规则执行顺序

执行顺序是确定的：
1. 先 `global_rules`。
2. 后 provider `rules`。
3. 各自内部按 `priority` 升序。
4. `priority` 相同，按条件数（`specificity`）升序。
5. 仍相同则按文件书写顺序。

结论：后执行规则更容易覆盖先执行结果。

## 5.3 逻辑能力与限制

支持：
- 单条规则内 `and`（默认）或 `or`。

不支持：
- 单条规则内括号嵌套（如 `A and B and (C or D)`）。

落地建议：
- 同字段“或”优先用 `in` 或 `regex`。
- 跨字段复杂“或”拆成多条规则。

## 5.4 条件操作符语义

| 操作符 | YAML 写法 | 语义 |
|---|---|---|
| `equals` | `equals: "支出"` | 完全相等 |
| `contains` | `contains: "交通"` | 包含子串 |
| `regex` | `regex: "(?i)coffee"` | 正则命中 |
| `starts_with` | `starts_with: "银行卡"` | 前缀匹配 |
| `ends_with` | `ends_with: "-TEST"` | 后缀匹配 |
| `in` | `in: ["余额支付", "银行卡支付"]` | 等于列表任一项 |
| `not_empty` | `not_empty:` | 字段存在且非空 |
| `is_empty` | `is_empty:` | 字段为空或不存在 |
| `greater_than` | `greater_than: 100` | 数值大于阈值 |
| `less_than` | `less_than: 100` | 数值小于阈值 |
| `between` | `between: { min: 50, max: 200 }` | `min <= value <= max` |

`not_empty` / `is_empty` 正确写法（注意是空值写法）：

```yaml
conditions:
  - field: "peerAccount"
    not_empty:
  - field: "status"
    is_empty:
```

## 5.5 action 字段与合并语义

`action` 支持：
- 账户类：`debit_account`、`credit_account`、`fee_account`、`pnl_account`、`rounding_account`
- 文本类：`payee`、`narration`
- 控制类：`ignore`、`flag`
- 集合类：`tags`、`links`
- KV 类：`metadata`

多规则命中时：
- 单值字段：后命中覆盖先命中。
- `metadata`：按 key 合并，后命中覆盖同 key。
- `tags/links`：追加，不自动去重。
- `ignore`：一旦为 `true` 保持 `true`。

## 5.6 `priority` / `terminal` / `ignore`

- `priority`：控制先后，不是“是否生效”。
- `terminal: true`：命中后立即停止后续规则。
- `ignore: true`：该记录最终不输出分录。

## 6. mapping 配置

典型文件：`src/mapping/<provider>.yml`

```yaml
date: "交易时间"
payee: "交易对方"
narration: "商品"
transaction_type: "收/支"
amount:
  column: "金额(元)"
  transform: abs

extra_fields:
  peer: "交易对方"
  peerAccount: "对方账号"
  orderId: "交易单号"
  txType: "交易类型"
  method: "支付方式"

date_formats:
  - "%Y-%m-%d %H:%M:%S"
  - "%Y/%m/%d %H:%M"
```

### 6.1 字段写法

- 简写：`payee: "交易对方"`
- 详写：

```yaml
payee:
  column: "交易对方"
  default: "匿名对手"
  transform: null
  regex_extract: "^(.*?)(\(|$)"
```

### 6.2 详写字段含义

- `column`：来源列名。
- `default`：字段为空时默认值。
- `transform`：常用 `abs`、`negate`（主要用于数值字段）。
- `regex_extract`：正则提取；优先返回第 1 捕获组，否则返回完整匹配。

### 6.3 `extra_fields` 兼容格式

- 推荐：`metadata_key -> csv_column`
- 兼容旧格式：`csv_column -> metadata_key`

### 6.4 XLSX 表头自动识别

当 `has_csv_header: true` 且输入为 `xlsx` 时，reader 会根据 mapping 的列名匹配得分自动选择最可能的表头行。

## 7. metadata key 归一化

输出前会做 metadata key 规范化：
- 常见中文键映射到统一英文键（如 `交易状态 -> status`、`交易对方 -> peer`）。
- 非法键会转换为合法 ASCII 标识。
- 无法转换时回退稳定哈希键（`meta_xxxxxxxx`）。

建议优先使用目标键名：
- `peer`、`peerAccount`、`source`、`txType`、`type`、`payTime`、`method`、`orderId`、`merchantId`

## 8. 账户回退优先级（含证券账户白话说明）

### 8.1 普通现金流回退顺序

支出：
1. `rules[].action.debit_account`
2. `default_expense_account`
3. `Expenses:Unknown`

支出对应资产账户：
1. `rules[].action.credit_account`
2. `default_asset_account`
3. provider 内置资产兜底

收入：
1. `rules[].action.credit_account`
2. `default_income_account`
3. `Income:Unknown`

### 8.2 证券账户分别在记什么

- `持仓账户`（如 `Assets:...:Securities`）：记录证券份额变化。
- `券商现金账户`（如 `Assets:...:Cash`）：记录券商资金账户现金变化。
- `fee_account`：手续费/交易规费。
- `pnl_account`：卖出时承接已实现盈亏。
- `rounding_account`：承接尾差/精度差。
- `Income:Investing:Interest`：逆回购到期利息（当前固定值）。
- `Assets:Transfer:Broker`：银证转账对手资产账户。

### 8.3 证券场景怎么用这些账户

1. 普通买入（spot buy）
- 持仓增加、券商现金减少。
- 差额 > 0 入 `fee_account`。
- 差额 < 0 入 `rounding_account`。

2. 普通卖出（spot sell）
- 持仓减少、券商现金增加。
- 费用差额入 `fee_account`。
- 追加一条 `pnl_account` 空金额分录，让 Beancount 平衡已实现盈亏。

3. 逆回购（repo）
- 固定面值：每份 `100 CNY`。
- 买入侧差额按 `fee/rounding` 处理。
- 到期侧：正差额记利息收入，负差额记费用损失。

4. 银证转账（cash transfer）
- 仅资产账户间划转，不使用 `pnl/rounding`。
- 原始 `fee/tax` 仅保留在 metadata。

### 8.4 `fee/pnl/rounding` 覆盖顺序（按代码行为）

`fee_account`：
1. `rules[].action.fee_account`
2. `default_fee_account`
3. `default_expense_account`
4. `Expenses:Investing:Fees`

`pnl_account`：
1. `rules[].action.pnl_account`
2. `default_pnl_account`
3. `default_income_account`（若是 `Income:Unknown` 会被忽略）
4. `Income:Investing:Capital-Gains`

`rounding_account`：
1. `rules[].action.rounding_account`
2. `default_rounding_account`
3. 从 `fee_account` 推导（最后一段替换为 `Rounding`）
4. `Expenses:Investing:Rounding`

补充：
- 逆回购利息账户当前固定为 `Income:Investing:Interest`。
- 银证转账默认对手资产账户为 `Assets:Transfer:Broker`。

## 9. 严格模式（`--strict`）

- 解析阶段：列数异常、映射错误会立即失败。
- 转换阶段：任意一条记录转换失败会立即退出。

默认（非 strict）下，坏行会警告并跳过，流程继续。

## 10. 常见误区与排查

- 误区1：把 `not_empty` 写成 `not_empty: true`。
  - 正解：写成 `not_empty:`。
- 误区2：以为 `priority` 越大越早执行。
  - 实际：`priority` 大的更晚执行。
- 误区3：以为 `tags/links` 自动去重。
  - 实际：只追加，不去重。
- 误区4：在单条规则里写括号嵌套表达式。
  - 实际：不支持，需拆规则或用 `in/regex`。

调试命令（示例）：

```bash
cargo run -- --provider alipay --source testsets/支付宝交易明细测试数据集.csv --config examples/alipay/advanced.yml --output tmp/output/debug.beancount --log-level info
```

重点看日志：
- `CSV/XLSX headers`
- `parsing complete`
- `Transformation complete: success / ignored / failed`
